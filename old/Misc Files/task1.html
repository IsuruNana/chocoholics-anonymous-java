GLOSSARY

Acme Accounting Services: Acme is a third-party organization and is responsible for financial procedures and reinstating suspended
members who have now paid what is owing. The Acme computer updates the relevant ChocAn Data Center computer membership records each evening at 9 P.M.

Choc An Data Center: Location where the Choc An Data is stored, organized and utilized

Electronic Funds Transfer (EFT): The payment made by the customer is done through EFT. Banking computers will later ensure that each provider’s bank account is credited with the appropriate amount.

Invalid Number: Customer number is not registered in system.

Member suspended: Customer owes money for the membership fees. Member is suspended.

Provider: Each health care professional who provides services to ChocAn members

Provider Directory: Lists the appropriate six-digit service code corresponding to the service provided.

Report: A list of services provided to that member, sorted in order of service date.
Providers also have a copy of the customers list as well as their own unique report

Terminal: System that calls Choc An Data Center after user swipes card.

Validated: Appears when a succesful swipe of the card is registered.

USE CASES

Alan and Robert

Use Case: Receive Health Care Services
Context: Provider enters data into system (simulated via keyboard input)
Actors: Provider and Member
Main Success Scenario:
     1. After provider switches terminal on, terminal prompts provider for 9-digit provider PIN (simulated via screen).
     2. System validates authentication.
     3. Member swipes their card through terminal’s card reader (simulated via keyboard input).
     4. System validates member.
     5. System ejects card.
     6. Provider passes member card again through scanner or enters member’s 9-digit PIN (simulated via keyboard input).
     7. System validates member.
     8. After validation, healthcare provider enters date of service according to the format “MM-DD-YYYY."
     9. Provider searches for correct 6-digit service code by accessing Provider Directory.
     10. Provider enters service code into terminal (simulated via keyboard input).
     11. System prints service name based on code (up to 20 characters simulated via screen) to provider’s terminal.
     12. System prompts provider to confirm service is correct (simulated via screen).
     13. Provider confirms selection and enters any additional comments regarding service (simulated via keyboard input).
     14. System writes a transaction record to disk with the following fields.
           Today’s Date and time (MM-DD-YYYY HH:MM:SS)
           Date of Service (MM-DD-YYYY)
           Provider PIN (9 digits)
           Member PIN (9 digits)
           Service code (6 digits)
           Additional (optional) Comments (100 characters)
     15. System bills ChocAn after healthcare service.
     16. System ejects card.
Extensions:
     8a. Invalid date format is entered.
           8a.1. System informs provider.
           8a.2. Provider enters correct format.
           8a.3. Use case continues at step 9.
     10a. An invalid service code was entered.
           10a.1. System informs provider.
           10a.2. Provider enters correct code.
           10a.3. Use case continues at step 11
     12a. Service code is not that which was requested by provider.
           12a.1. Provider enters correct service code.
           12a.2. Use case continues at step 11.


Use Case: Validation
Context: Member or provider must interact with provider terminal and be validated.
Actors: Member or Provider and ChocAn Data Center Computer
Main Success Scenario:
     1. After provider terminal reads card or PIN, it dials ChocAn Data Center.
     2. ChocAn Data Center Computer verifies member’s PIN.
           2.1. ChocAn Data Center Computer communicates with provider terminal to
                   print “Validated” on terminal’s one-line display.
Extensions:
     1a. Invalid provider PIN was entered.
           1a.1. System informs provider and member (simulated via screen) as to
                  reason for invalid entry by printing message such as
                 “Invalid number,” “Member suspended,” etc.

     3. Provider enters date and time of transaction, current date and time, service code, member’s name and PIN, and fee on
         another verification form.

     None

Use Case: Creation of an EFT File
Context: Creation of a list of funds to be transferred to the bank account of a provider. This is automatically done every Friday at midnight.
Actors: Time, provider
Main Success Scenario:
1.      Every Friday at midnight, generates an EFT file containing the following for a provider:
Provider name and number
Amount to be transferred to the provider’s account
1.1.   The amount to be transferred to the provider’s account is the sum of the cost of all services done by the provider in the past week according to the Bill ChocAn for Healthcare Services

2.      Step 1 repeats for each provider each Friday at midnight.
Extensions:
          None

Name (25 characters)
Number (9 digits)
Address (25 characters)
City (14 characters)
State (2 letters)
ZIP Code (5 digits)
2.      For each service provided, the report then includes the following:
Date of service (MM-DD-YYYY)
Date and time data were received by the computer (MM-DD-YYYY HH:MM:SS)
Member name (25 characters)
Member number (9 digits)
Service code (6 digits)
Fee to be paid (up to $999.99)

Total number of consultations with members (3 digits)
Total fee for week (Up to $99,999.99)
3.1.   These stats are found/created by accessing the record from Bill ChocAn for Health Services for the provider
4.      Step 1-3 repeats for each member each Friday at midnight.
